# Evening Call Summary — DA2

The evening discussion helped separate two things I had been blending together: a model that actually chooses tools and a pipeline where Python already decided the sequence and the model only fills one slot inside it. Samuel walked through TheConversionEngine step by step and showed that the planning lives in the orchestrator, not in the model, because the tool calls are sequenced directly in code and the generation call never receives tool schemas in the first place. What made the explanation land was his three-pattern distinction — model-as-planner, orchestrator-as-planner with model-as-transform, and model-as-reviewer — because it gave us a simple way to place the generation layer and the governance layer without overclaiming what the system is doing. We ended the call with a concrete wording change for `method.md` and `README.md`: describe the system as orchestrator-driven, with the model rewriting content and influencing one governance gate, rather than saying the model itself is using tools.
