# Sign-off — DA2

Gap status: closed.

Before this explainer I could not give a clear account of where the model's contribution ends and the orchestrator's begins in a system like TheConversionEngine. I described the system as a "sales agent with tools" without being able to defend whether that was the model choosing tools or scaffolding code executing them on a fixed schedule. The explainer my partner wrote made the distinction concrete: there is a meaningful difference between the model selecting and executing tools via native function calling, the model producing structured output that the orchestrator interprets, and the orchestrator running deterministic tool sequences while the model only rewrites or reviews content. In TheConversionEngine, the toolchain sequence in agent/orchestration/service.py is fully deterministic scaffolding. The model appears in generation_service.draft_email_from_scaffold to rewrite a draft, and in the Week 11 judge to review a candidate action. Those are real model contributions, but they are not tool selection.

What I understand now that I did not before: saying a system "uses tools" correctly describes a system where the model has a tool schema and chooses which tools to call and in what order based on context. It does not correctly describe a system where code decides the sequence and the model rewrites within a fixed scaffold. The Week 10 proactive tool-use instruction mattered precisely because it tried to add a model-led component to a scaffolded system, and the gap in my understanding was that I could not distinguish those two control patterns in my own architecture description.
