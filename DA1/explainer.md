# What Actually Gets Faster in a Sales Agent?

This question came out of a pretty specific gap in my Week 10 and Week 11 work. In TheConversionEngine, the outreach flow in agent/policies/service.py builds a draft email by passing a lot of repeated structured context into the email generation step. That context includes segment labels, signal summaries, confidence-by-signal entries, top-quartile practices, safe-gap framing, and do-not-claim rules. Then in the evaluation artifacts, held_out_traces.jsonl and method.md report latency, token counts, cost, and the number of model calls. The part I could not explain cleanly was this: how much of that time was the model reading the prompt, how much was the model generating the answer, and when repeated prompt content was actually being reused instead of recomputed.

The first split I needed to understand was prefill versus decode. Prefill is the model reading the prompt and building the internal state it needs. Decode is the model generating new tokens one step at a time. For a sales agent, that split matters because the prompt often carries a lot of repeated instruction text, policy framing, and background context, while the response is short. So if I keep sending a large stable prompt prefix on every request, most of the waste shows up in prefill. Decode still matters, but it usually is not the main cost when the output is just a short outreach draft.

The second split was KV cache versus prefix caching. KV cache is the model-side memory of prior tokens inside one generation. Once the model has processed earlier tokens, it stores the attention keys and values so it does not have to recompute them at every next-token step. That is what makes autoregressive decoding practical. Systems such as vLLM's PagedAttention are about managing that KV cache efficiently in memory so the server can keep more active sequences without wasting GPU space. Prefix caching is a different issue. It asks whether a serving layer or provider can reuse computation for the same prompt prefix across separate requests. So KV cache explains why the model does not keep rereading the same prefix inside one generation, while prefix caching explains when a repeated prefix across requests can avoid a fresh prefill cost.

The catch is that reuse only works when the shared prefix is actually stable. Anthropic's prompt caching docs make this very concrete: the cache is keyed off the exact prompt prefix up to a cache breakpoint, and changes to earlier blocks invalidate that cached prefix. Their docs also make the hierarchy explicit: tools, then system, then messages. So if tool definitions change, everything after them is invalidated. If system content changes, the system and message portion get invalidated. If message content changes, the message portion gets invalidated. The same docs also call out easy ways to break reuse by accident: putting the cache breakpoint on content that changes every request, adding timestamps or other dynamic metadata, moving the breakpoint too far from the last prior write, or serializing structured content in unstable key order. That was the part I had been hand-waving. "Mostly the same" is not enough. The reusable part has to stay the same at the token level.

That is exactly where this matters for TheConversionEngine. Some parts of my prompt assembly are good candidates for a stable prefix: fixed policy instructions, style constraints, reusable framing rules, and any consistent scaffold text that does not vary by prospect. Other parts are clearly volatile: the strongest-signal summary, the per-prospect confidence details, peer-specific notes, and the exact ask. If those volatile pieces get mixed early into the same block as the stable policy text, then even a small change can force the system to pay the full prefill cost again. If instead the stable policy block stays deterministic and the volatile prospect-specific details are appended later, the request becomes much more cache-friendly. That does not guarantee a provider-level cache hit by itself, but it is the prerequisite for one. That is the difference between “this prompt has repeated content” and “this system can actually reuse the repeated part.”

This also changes how I should read my Week 11 traces. If a row in held_out_traces.jsonl shows higher prompt-token counts, more model calls, and worse latency, I should not just shrug and say "the model was slower." A better explanation is that the system likely paid more prompt-ingestion cost, possibly multiple times, before it ever got to a short decode phase. So my benchmark write-up in method.md needs a cleaner interpretation of latency and cost: not just how many tokens were used, but which tokens were stable repeated prefix, which tokens were volatile suffix, and whether the serving path had any real chance to reuse earlier computation.

The design rule I take away is pretty straightforward. Keep the stable prefix truly stable. Serialize structured fields deterministically. Do not put timestamps, trace metadata, or other per-request noise into the reusable prefix. Put the last stable shared block before the volatile prospect-specific content. And when I claim a caching or latency improvement, I need to measure it in a way that separates one-call versus two-call flows and repeated-prefix behavior instead of treating all input tokens as equivalent. If I cannot identify which tokens are stable, which are volatile, and where prefill dominates cost, then I cannot make reliable latency or cost claims about an agent system.
